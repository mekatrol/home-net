# Architecture

This service is the `ai` home monitor container. It watches Raspberry Pi and
home-network devices, publishes status, runs scheduled maintenance commands,
and exposes a small web UI for email redirect management.

## Runtime Shape

- `Dockerfile` builds an Ubuntu image with Python, SSH, Supervisor, and the app
  copied into `/opt/monitor`.
- `/start.sh`, generated by the Dockerfile, starts SSH, fixes ownership on
  mounted volumes, ensures the WebSocket TLS certificate exists, and runs
  `app/main.py` under Supervisor.
- `create.sh` builds and runs the `ai:latest` container on the `docker-network`
  ipvlan network. Runtime config is mounted from `./run/config`, secrets from
  `./run/secrets`, logs from `./logs`, and email storage from `./emails`.
- `scripts/deploy.sh` installs the remote watchdog command dispatcher and MQTT
  ping service onto monitored Raspberry Pi devices.

## Backend Entry Point

`app/main.py` is the coordinator. It loads `/run/config/config.yaml`, falls
back to `app/config.yaml` when needed, then starts long-running asyncio tasks:

- WebSocket device server on `server.host:server.port`.
- MQTT bridge, MQTT heartbeat listener, and retained MQTT status publisher.
- HTTP pollers for devices configured with `http_url`.
- Watchdog silence detector and scheduled upgrade/reboot loops.
- aiohttp web server for redirect and dropped-email administration.
- Email polling, processing, sending, and cleanup workers when
  `email_config.yaml` exists.

Keep new background work as an explicit task from `main.py` so startup behavior
stays visible.

## Device Monitoring

Device config is modeled in `app/watchdog_models.py`.

- WebSocket devices have a `device_name` and connect to `WatchdogServer`.
  They can receive admin commands such as `reboot`, `upgrade`,
  `upgrade_reboot`, and `restart_container`.
- MQTT-only devices are marked by `mqtt_topic`. The server subscribes to the
  topic and treats messages as heartbeats. These devices cannot be rebooted by
  the monitor.
- HTTP-polled devices are marked by `http_url`. The server polls the URL and
  treats healthy responses as heartbeats.
- `DeviceState` is the shared in-memory state for last-seen time, wall-clock
  timestamps, pending commands, cooldowns, and WebSocket connection status.

`app/watchdog_schedulers.py` owns the silence loop and scheduled maintenance.
Silence is measured with `time.monotonic()` against
`miss_threshold * ping_interval`. WebSocket devices can be rebooted; MQTT-only
and HTTP-polled devices log warnings because they have no command channel.

## WebSocket Command Path

`app/watchdog_server.py` handles two WebSocket auth modes:

- Device auth: first message is `{"type": "auth", ...}`. After auth, heartbeat,
  log, and command-result messages update `DeviceState`.
- Admin auth: first message is `{"type": "admin_auth", ...}`. Admin clients can
  list devices and send commands from a strict allowlist.

Command sending lives in `app/watchdog_commands.py`. Scheduled and admin
commands flow through this module instead of writing to WebSockets directly.

## MQTT Path

`app/watchdog_mqtt.py` wraps paho MQTT in `MqttBridge`. The paho client runs in
its own thread and hands messages into the asyncio loop through a queue.

Status is published to `status/{mqtt_device_name}` with retained messages.
Payloads include `lastStatus`, `lastStatusTimestamp`, and
`lastOnlineTimestamp`.

## Web UI And API

`app/watchdog_web.py` serves both JSON APIs and the compiled Vue app from
`app/web`.

API routes:

- `GET /api/health`
- `GET /api/redirects`
- `PUT /api/redirects`
- `GET /api/dropped-emails`
- `POST /api/dropped-emails/delete`
- `POST /api/dropped-emails/continue`

Authentication is a bearer token or `X-Admin-Token` matching `web.web_pwd`.
Redirect writes go through `RedirectConfigStore`, which serializes access with
an asyncio lock and saves back into `email_config.yaml`.

The frontend source is in `web-ui/`. It is a Vue, Vite, TypeScript, Pinia, and
Vue Router app. `web-ui/src/stores/redirects.ts` owns auth token storage,
redirect API calls, dropped-email API calls, and UI state. Build output is
checked into `app/web` for the backend container to serve.

## Email Pipeline

Email config is loaded from `/run/config/email_config.yaml` by
`load_email_config()`.

`app/watchdog_email.py` owns the file-based pipeline:

- POP3 polling fetches mail and writes `.eml` files into the inbox tree.
- Inbox and processing workers move messages through staged directories.
- Processed mail is sent through SMTP.
- Sent, dropped, and orphaned metadata cleanup loops enforce retention.
- Dropped mail can be deleted or resumed from the web UI.

Processors live in `app/processors/` and run in this order:

1. Drop detector.
2. Spam detector.
3. Redirection detector.

Processors mutate a shared context dict. If a processor returns `False`, the
chain stops and the email lands in the context-selected destination directory.
Metadata sidecars use `.eml.meta.json` and are written beside the email file.

## Config Files

- `app/config.example.yaml` documents server, web, MQTT, logging, status, and
  device options.
- `/run/config/config.yaml` is the real runtime monitor config.
- `/run/config/email_config.yaml` is the real runtime email and redirect config.
- `app/watchdog_redirects.py` normalizes redirect rules and preserves the rest
  of the email config when saving redirects.

## Remote Device Agent

`remote/main.py` is the lightweight monitored-device service. Deployment
scripts install it as the `ping` service on remote Pis. The deploy flow also
installs a restricted `watchdog` SSH user and command dispatcher so monitor
commands remain allowlisted.

## Test And Build Commands

- Backend dependencies: `app/requirements.txt`
- Remote agent dependencies: `remote/requirements.txt`
- Frontend install/build/test commands live in `web-ui/package.json`
- Frontend build: `npm run build` from `web-ui/`
- Frontend unit tests: `npm run test:unit` from `web-ui/`
- Frontend lint: `npm run lint` from `web-ui/`
